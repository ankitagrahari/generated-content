The Mental Model

Before writing a single line, internalize these 4 mental shifts. Everything else follows from them.

🌊

Everything is a stream

A click, an API call, a DB query, a list — all are Observable<T>. Stop thinking "call this function". Think "subscribe to this event source".

🔗

Operators transform, not execute

map(), filter(), flatMap() define HOW data flows. Nothing runs until subscribe(). The chain is a blueprint, not execution.

🧵

Threads are declared, not managed

You never start a Thread. You declare: "produce on IO thread, consume on Main". subscribeOn() and observeOn() handle the rest.

💥

Errors are first-class citizens

onError is a terminal event like onComplete. Handle errors in the chain with onErrorReturn, onErrorResumeNext — not try/catch.

The 5 questions to ask before writing a chain
Q1 What is my source? (API call? DB? User event? In-memory list?)
→ Single for one item, Observable for many, Completable for no data, Maybe for 0 or 1
Q2 Does each emission spawn another async operation?
→ Yes → flatMap/switchMap/concatMap. No → map/filter
Q3 Do I need to combine multiple streams?
→ Both must emit → zip. Latest from each → combineLatest. One then other → concat/flatMap
Q4 Where should work happen, where should result land?
→ subscribeOn(Schedulers.io()) for work, observeOn(AndroidSchedulers.mainThread()) for UI
Q5 What happens when it fails?
→ Retry? → retry(n). Fallback value? → onErrorReturn. Alternative stream? → onErrorResumeNext

Real-World Scenarios

Each scenario follows the same structure: Problem → Think → Build → Pitfalls.

01

Search with Debounce + API Call

User types in a search box. Call API 300ms after they stop typing. Cancel previous call if they type again.

▼
ThinkSource: text change events (Observable<String>)
Each text → async API call → need switchMap (cancels previous)
Java · RxJava 2/3Search Scenario
// UI layer gives us an Observable of text changes
RxTextView.textChanges(searchEditText)          // Observable<CharSequence>
    .debounce(300, TimeUnit.MILLISECONDS)        // wait 300ms after last keystroke
    .map(CharSequence::toString)                // convert to String
    .filter(query -> query.length() >= 3)       // skip short queries
    .distinctUntilChanged()                     // skip if same as last
    .switchMap(query ->                         // KEY: cancels prev call!
        apiService.search(query)                // Single<List<Result>>
            .toObservable()
            .onErrorResumeNext(err ->           // don't kill stream on error
                Observable.just(Collections.emptyList())
            )
    )
    .subscribeOn(Schedulers.io())
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe(
        results -> adapter.submitList(results),
        error -> showError(error)
    );
⚠ Common Mistake

Using flatMap instead of switchMap. flatMap lets ALL calls run concurrently — results arrive out of order. switchMap disposes the previous inner observable when a new item arrives. Always use switchMap for "latest query wins" semantics.

✓ Pro Tip

Store the Disposable! disposable = observable.subscribe(...); and call disposable.dispose() in onDestroy/onPause. Without this, you leak subscriptions.

02

Parallel API Calls → Combine Results

Fetch user profile + user orders simultaneously, merge them into a single UI model.

▼
ThinkTwo independent sources. Both must emit before we proceed. → zip(). Run both on io thread.
Java · RxJava 2/3Parallel + Zip
Single<UserProfile> profileSingle =
    apiService.getUserProfile(userId)
        .subscribeOn(Schedulers.io());          // runs on IO thread

Single<List<Order>> ordersSingle =
    apiService.getUserOrders(userId)
        .subscribeOn(Schedulers.io());          // also runs on IO thread (parallel!)

Single.zip(profileSingle, ordersSingle,     // waits for BOTH
    (profile, orders) ->
        new UserDashboard(profile, orders)        // combine
)
.observeOn(AndroidSchedulers.mainThread())
.subscribe(
    dashboard -> renderDashboard(dashboard),
    error -> showError(error)
);
✓ Why this is parallel

Each Single has its own subscribeOn(io). When zip subscribes to both, they fire simultaneously on separate IO threads. zip() collects both results and emits only when both arrive. No thread blocking.

JavaVariant: 3+ calls with zipArray
// When you have dynamic number of API calls
List<Single<ItemDetail>> calls = ids.stream()
    .map(id -> apiService.getItem(id).subscribeOn(Schedulers.io()))
    .collect(Collectors.toList());

Single.zip(calls, results -> {
    List<ItemDetail> items = new ArrayList<>();
    for (Object r : results) items.add((ItemDetail) r);
    return items;
}).subscribe(items -> renderList(items));
03

Sequential Dependent Calls (Chain)

Get user → use userId to get their team → use teamId to get team members. Each call depends on the previous result.

▼
ThinkSequential, dependent → flatMap chains. Each step returns a new Single. The chain reads like a story: "get user, then get team, then get members".
Java · RxJava 2/3flatMap chaining
apiService.getCurrentUser() // Single<User> .flatMap(user -> apiService.getTeam(user.getTeamId()) // Single<Team> ) .flatMap(team -> apiService.getTeamMembers(team.getId()) // Single<List<Member>> ) .subscribeOn(Schedulers.io()) .observeOn(AndroidSchedulers.mainThread()) .subscribe( members -> showMembers(members), error -> handleError(error) ); // ────── Variant: carry context forward ────── // If you need BOTH user AND members in the result: apiService.getCurrentUser() .flatMap(user -> apiService.getTeam(user.getTeamId()) .flatMap(team -> apiService.getTeamMembers(team.getId()) .map(members -> new TeamScreen(user, team, members)) ) ) .subscribe(screen -> render(screen));
⚠ Don't nest too deep

If you're nesting flatMaps 4+ levels deep, extract each step into a private method that returns a Single/Observable. Flat chains read better than nested pyramids.

04

Cache → Network with Fallback

Show cached data immediately if available, then fetch fresh data. If network fails, show cache instead of error.

▼
ThinkTwo sources. Want BOTH to emit (cache first, then network). Use concat() or merge() strategically. For error fallback: onErrorResumeNext.
JavaCache + Network Strategy
// Strategy 1: Cache first, then network (show updates) Observable<List<Article>> cacheSource = database.getArticles() // Maybe or Observable .toObservable() .onErrorResumeNext(Observable.empty()); // skip if cache empty Observable<List<Article>> networkSource = apiService.fetchArticles() .toObservable() .doOnNext(articles -> database.saveArticles(articles) // save fresh data .subscribeOn(Schedulers.io()) .subscribe() ) .onErrorResumeNext(err -> database.getArticles().toObservable() // fallback to cache ); Observable.concat(cacheSource, networkSource) // cache first, then network .distinctUntilChanged() // skip if same data .subscribeOn(Schedulers.io()) .observeOn(AndroidSchedulers.mainThread()) .subscribe(articles -> updateUI(articles));
✓ concat vs merge

concat runs sources sequentially (waits for cache to complete before network). merge subscribes to both simultaneously — use when both sources are independent and you want the fastest to win.

05

Polling with Retry & Exponential Backoff

Poll an endpoint every 5s. On failure, retry up to 3 times with exponential backoff (1s, 2s, 4s delay).

▼
JavaPolling + Exponential Backoff
Observable.interval(0, 5, TimeUnit.SECONDS, Schedulers.io()) .flatMap(tick -> apiService.checkStatus() .toObservable() .retryWhen(errors -> errors.zipWith( Observable.range(1, 3), // max 3 retries (error, attempt) -> attempt ) .flatMap(attempt -> Observable.timer( (long) Math.pow(2, attempt - 1), // 1s, 2s, 4s TimeUnit.SECONDS ) ) ) .onErrorReturn(e -> Status.error(e)) // don't kill outer stream ) .distinctUntilChanged() .observeOn(AndroidSchedulers.mainThread()) .subscribe(status -> updateStatusUI(status));
⚠ Critical: onErrorReturn inside flatMap

Without onErrorReturn/onErrorResumeNext INSIDE the flatMap, a single failed poll will terminate the entire outer interval stream. Always handle inner stream errors locally so the outer polling loop keeps running.

06

Rate Limiting / Batch Processing

Given 1000 IDs, fetch in batches of 50. Don't flood the API.

▼
JavaBatching with buffer + concatMap
Observable.fromIterable(allIds) // 1000 IDs .buffer(50) // groups into List<String> of 50 .concatMap(batch -> // sequential, not parallel apiService.fetchBatch(batch) .toObservable() .delay(200, TimeUnit.MILLISECONDS) // throttle between batches ) .subscribeOn(Schedulers.io()) .subscribe( results -> processResults(results), error -> handleError(error), () -> onAllDone() ); // concatMap = sequential flatMap // flatMap = parallel (all at once, risky) // concatMap = one batch finishes, then next starts

How to Build Long Chains

The trick: think in layers. Each layer has a job. Write them one at a time, test as you go.

The 6-Layer Framework
1

Source Layer — What emits?

Observable.fromCallable() / Single.fromCallable() / RxTextView.textChanges() / PublishSubject
Don't call the function — wrap it: fromCallable(() -> blockingCall())

2

Transform Layer — Shape the data

map(), filter(), flatMap(), switchMap(), concatMap()
Ask: "does this transform synchronously?" → map. "does it create another async source?" → flatMap

3

Combine Layer — Multiple sources?

zip(), combineLatest(), merge(), concat(), amb()
Only add this layer if you have 2+ streams to join

4

Control Layer — Timing & flow control

debounce(), throttleFirst(), buffer(), window(), take(), skip()
Add only for UI events or high-frequency streams

5

Error Layer — What happens on failure?

onErrorReturn(default), onErrorResumeNext(fallbackStream), retry(n), retryWhen(handler)
Always handle errors! Unhandled = crash

6

Thread Layer — Where does it run?

subscribeOn(Schedulers.io()) — once per chain
observeOn(mainThread()) — right before UI update
Additional observeOn() to hop threads mid-chain

flatMap vs switchMap vs concatMap — The Decision Tree
Decision Guide
// Question: multiple inner observables from each item? // ✓ Use flatMap when: // - Order doesn't matter // - All results needed (e.g. save multiple items simultaneously) // - No new item invalidates previous items.flatMap(item -> save(item)) // all saves run in parallel // ✓ Use switchMap when: // - Latest item cancels previous (search, autocomplete) // - Only the LAST emission's result matters searchText.switchMap(q -> api.search(q)) // old request cancelled // ✓ Use concatMap when: // - Order matters (process 1, then 2, then 3) // - Don't want parallel execution // - Sequential queue behavior actions.concatMap(action -> processAction(action))
Avoiding callback hell — before and after
❌ Before RxJava — callback hell
api.getUser(userId, new Callback<User>() { public void onSuccess(User user) { api.getOrders(user.getId(), new Callback<Orders>() { public void onSuccess(Orders orders) { api.getRecommendations(orders, new Callback<List>() { // impossible to read, impossible to test }); } }); } });
✓ After RxJava — readable chain
apiService.getUser(userId) .flatMap(user -> apiService.getOrders(user.getId())) .flatMap(orders -> apiService.getRecommendations(orders)) .subscribeOn(Schedulers.io()) .observeOn(AndroidSchedulers.mainThread()) .subscribe(recs -> showRecommendations(recs), err -> showError(err));

Threading Without Blocking

The golden rule: never call .blockingGet() or .blockingFirst() on the main thread. Use schedulers instead.

Scheduler Reference
SchedulerUse ForThread Pool
Schedulers.io()Network, disk I/O, DB queriesUnbounded cached thread pool
Schedulers.computation()CPU-heavy work (parsing, sorting)Fixed (# of CPU cores)
Schedulers.newThread()Isolated one-off workNew thread per subscribe
Schedulers.single()Sequential work, no parallelismSingle background thread
Schedulers.trampoline()Testing, queued on current threadCurrent thread (queued)
AndroidSchedulers.mainThread()UI updates (Android)Android main/UI thread
The subscribeOn vs observeOn pattern
JavaThread hopping mid-chain
// subscribeOn: where the SOURCE subscribes (set once) // observeOn: downstream operators run on this thread // Example: Process on IO, heavy transform on Computation, show on Main apiService.fetchData() // starts on IO thread .subscribeOn(Schedulers.io()) // IO thread: subscription + fetch .map(rawData -> parseJson(rawData)) // still on IO .observeOn(Schedulers.computation()) // hop to computation .map(parsed -> heavyTransform(parsed)) // on computation thread .observeOn(AndroidSchedulers.mainThread()) // hop to main thread .subscribe(result -> textView.setText(result)); // UI update on main // RULE: multiple observeOn() allowed, but subscribeOn() only matters FIRST time it appears
✓ Never block the stream

If you need a blocking call inside a chain, wrap it:
Single.fromCallable(() -> blockingDatabaseQuery())
This wraps the blocking call in a lazy supplier and runs it on the subscribed scheduler — no thread blocked by RxJava itself.

⚠ The subscribeOn trap

subscribeOn() only affects the thread where the source STARTS. If you put it at the bottom of a chain with many flatMaps, inner observables may have already subscribed on default threads. Always put subscribeOn() right after the source, and use individual .subscribeOn() in inner Observables for parallel flatMaps.

CompositeDisposable — No more leaks
Java · Android ViewModel pattern
public class UserViewModel extends ViewModel { private final CompositeDisposable disposables = new CompositeDisposable(); public void loadUser(String id) { disposables.add( // track it apiService.getUser(id) .subscribeOn(Schedulers.io()) .observeOn(AndroidSchedulers.mainThread()) .subscribe(user -> liveData.setValue(user), err -> error.setValue(err)) ); } // Called automatically when ViewModel is cleared @Override protected void onCleared() { disposables.clear(); // disposes ALL at once } }

Operator Mental Map

The right operator for the right job. Organized by what problem they solve.

Transform
Transform
map(f)
Synchronous 1-to-1 transform. "Convert each item."
Transform
flatMap(f)
Each item → new Observable. Merge results. ORDER NOT GUARANTEED. Use for parallel async.
Transform
concatMap(f)
Like flatMap but sequential. ORDER GUARANTEED. Use when sequence matters.
Transform
switchMap(f)
Each new item cancels previous inner observable. Use for "latest wins" (search, tabs).
Transform
scan(acc, f)
Accumulate state over emissions. Like reduce but emits each step. Good for running totals.
Filter
Filter
filter(predicate)
Drop items that don't match. Most common gatekeeper.
Filter
distinctUntilChanged()
Skip if same as previous. Great for state streams (don't re-render identical state).
Filter
debounce(t)
Emit only after silence for t. Use for search inputs, form validation.
Filter
throttleFirst(t)
Emit first item, ignore for t. Use for button clicks (prevent double-click).
Filter
take(n) / skip(n)
Limit or skip emissions. take(1) completes after first — great for "fire once".
Combine
Combine
zip(s1, s2, f)
Pair items 1-to-1 from both. Both must emit. Use for parallel calls.
Combine
combineLatest(s1, s2, f)
Emit when EITHER emits, using latest from each. Use for form validation (username + password).
Combine
merge(s1, s2)
Interleave emissions from multiple streams. Use when both sources are active simultaneously.
Combine
concat(s1, s2)
s1 completes, then s2 starts. Sequential. Use for cache → network.
Combine
withLatestFrom(other)
When source emits, pair with latest from 'other'. Use for button click + latest form state.
Error Handling
Error
onErrorReturn(val)
Replace error with a default value. Stream completes normally.
Error
onErrorResumeNext(obs)
Replace error with another Observable. Great for fallback to cache.
Error
retry(n)
Resubscribe up to n times on error. Good for transient network failures.
Error
retryWhen(handler)
Custom retry logic (exponential backoff, conditional). Most powerful error recovery.
